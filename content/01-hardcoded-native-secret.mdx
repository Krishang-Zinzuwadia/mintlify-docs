## Challenge Summary
---

The challenge was an Android APK named `fam-ctf.apk`.

Prompt:

```text Output
Many times devs put secrets in code and forget to remove.

can you find that secret ??
```

The prompt strongly suggests a hardcoded secret. For Android APKs, that secret can be in resources, DEX bytecode, or native shared libraries.

## Recon
---


An APK is a ZIP archive, so I listed its contents without needing Android-specific tooling:

```powershell Terminal
tar -tf fam-ctf.apk
```

Important entries:

```text Output
classes.dex
classes2.dex
classes3.dex
classes4.dex
classes5.dex
lib/arm64-v8a/libfam.so
lib/x86_64/libfam.so
res/layout/activity_main.xml
resources.arsc
```

The key finding here was the pair of native libraries:

```text Output
lib/arm64-v8a/libfam.so
lib/x86_64/libfam.so
```

That matters because Android apps often call C/C++ code through JNI, and hardcoded native string constants frequently remain visible in the compiled `.so` file.

## Extracting The APK
---

I extracted the archive:

```powershell Terminal
mkdir apk_extracted
tar -xf fam-ctf.apk -C apk_extracted
```

Then I inspected the largest and most relevant extracted files:

```powershell Terminal
Get-ChildItem -Recurse apk_extracted |
  Select-Object FullName,Length |
  Sort-Object Length -Descending |
  Select-Object -First 30
```

This confirmed the app contained multiple DEX files and two architecture-specific copies of `libfam.so`.

## String Search
---

Since the challenge specifically mentioned developers leaving secrets in code, I searched printable strings across the extracted files for likely keywords:

```text Output
flag
ctf
secret
password
token
key
fam
```

The useful hits came from both native libraries:

```text Output
apk_extracted\lib\arm64-v8a\libfam.so
apk_extracted\lib\x86_64\libfam.so
```

The extracted strings included:

```text Output
Java_com_ctf_fam_MainActivity_getSecretFromNative
Java_com_ctf_fam_MainActivity_computeSignature
Java_com_ctf_fam_MainActivity_getDebugToken
Java_com_ctf_fam_MainActivity_getUsernames
libfam.so
FAM{str1ngs_d0nt_l13_1n_n4t1v3_l4nd}
```

The JNI symbol name is especially important:

```text Output
Java_com_ctf_fam_MainActivity_getSecretFromNative
```

This indicates the app exposes a native method from `MainActivity`, likely named `getSecretFromNative()`. The flag was compiled into the same native library and left as a plaintext string.

## Why This Worked
---

Moving a secret from Kotlin or Java into native C/C++ does not automatically hide it. If the native code contains a normal string literal, the compiler usually places that string in the binary's read-only data section.

Static string extraction can recover those literals without running the APK, installing it on a device, or decompiling the Java code.

The flag itself confirms the intended solve path:

```text Output
str1ngs_d0nt_l13_1n_n4t1v3_l4nd
```

In other words, strings still expose secrets in native land.

## Final High-Level Overview
---

1. Confirmed the APK was present.
2. Treated the APK as a ZIP archive and listed its contents.
3. Noticed the app included native libraries named `libfam.so`.
4. Extracted the APK.
5. Searched printable strings across the extracted files.
6. Found a JNI method named `getSecretFromNative`.
7. Recovered the plaintext flag from `libfam.so`.
